**Date:** 2026-05-30

**Scope:** Full codebase ŌĆö API routes, payment libs, auth, middleware, public endpoints, client components

## Executive Summary

The codebase is structurally sound for a launch-stage D2C store. The most important payment-integrity controls (server-side price recomputation, HMAC signature verification, COD/INTL gating) are all present. However there are **two blockers that will prevent real money from flowing** before they are fixed: the admin login infinite redirect loop locks out the admin entirely, and the `shippingAddress` type mismatch in the Zod schema causes every checkout with an address to return 400. Additionally, there are three High-severity issues that could be exploited in production: the `timingSafeEqual` crash on malformed signatures, the Stripe webhook bypass when `STRIPE_WEBHOOK_SECRET` is unset, and missing idempotency on `markPaid`.

**Overall Risk Rating: HIGH** (two pre-launch blockers + three exploitable highs; no existential secret-exposure or SQL injection found)

## Prioritised Remediation List

Fix in this order before running Meta ads at scale:

### Must-fix before any real traffic (Blockers)

1. **[F-01] Admin login infinite redirect loop** ŌĆö Admin cannot manage orders at all.
2. **[F-02] `shippingAddress` type mismatch** ŌĆö No order with an address can complete checkout.

### Fix before enabling real payment keys

3. **[F-03] `timingSafeEqual` crash on wrong-length signature** ŌĆö Signature attacks crash handlers with 500 instead of 400.
4. **[F-06] `providerOrderId = null` mismatch bypass** ŌĆö Any user with a valid Razorpay payment can mark a WhatsApp-flow order as paid.
5. **[F-04] Stripe webhook bypass when secret is unset** ŌĆö If `STRIPE_WEBHOOK_SECRET` not configured, webhook endpoint is wide open and payments won't be marked paid.
6. **[F-05] Missing idempotency in `markPaid`** ŌĆö Webhook retries can overwrite shipped-order state.

### Fix before scaling ad spend (High-traffic risks)

7. **[F-08] Webhook order lookup capped at 200** ŌĆö Switch to direct DB lookup by `providerOrderId`.
8. **[F-07] No rate limit on `/api/checkout` and `/api/verify`** ŌĆö Gateway abuse and brute-force vectors.
9. **[F-10] Missing Content-Security-Policy** ŌĆö XSS mitigation layer absent; needed before adding any user-generated content.
10. **[F-13] No `qty` or `items` array max** ŌĆö Prevents Razorpay overflow 500 and keeps checkout DoS cost high.

### Harden before high GMV

11. **[F-11] `Math.random()` order codes + unrate-limited tracking** ŌĆö Add `crypto.randomBytes` + rate limit on `/api/orders`.
12. **[F-09] Geo header spoofable outside Vercel** ŌĆö Document the Vercel-only assumption.
13. **[F-12] In-memory rate limiter ineffective on serverless** ŌĆö Move to Redis/Neon if brute-force is a concern.
14. **[F-14] Password length timing side-channel** ŌĆö Low practical risk; use bcrypt/argon2 for future peace of mind.
15. **[F-15] Razorpay Key ID duplication** ŌĆö Minor cleanup.
16. **[F-16] JsonLd `dangerouslySetInnerHTML` future risk** ŌĆö Add comment guard.
